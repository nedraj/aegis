## What is Aegis?

Aegis solves the **"last mile"** problem of AI: getting a real, useful large language model running in places with **zero or highly restricted internet**.

- **Model**: Phi-3-Mini-4k-Instruct (3.8B) — excellent reasoning per watt
- **Inference**: Ollama (easy air-gap model loading)
- **Orchestration**: K3s (lightweight Kubernetes)
- **Control Plane**: Custom "Mission Control" FastAPI service that **never** talks to OpenAI/Gemini
- **Transfer**: Single `.bundle` file containing images + weights + checksums

The system is **profile-driven** and **deterministic** — the same profile always produces the same deployment artifacts.

## Quick Local Start (5 minutes)

This spins up real Ollama + Mission Control on your laptop (no GPU needed).

```bash
git clone <this-repo>
cd deploy

make build
make test-local          # uses examples/docker-compose.local.yml
```

In another terminal:

```bash
curl -X POST http://localhost:8080/query \
  -H "Content-Type: application/json" \
  -d '{"prompt":"Give a one-sentence status of the forward sensor array."}'
```

You will receive a response that begins with **"MISSION UPDATE:"** generated by Phi-3 running locally.

## Full GCP Demo — Phase 3 vs Phase 4

### Phase 3 Flow (works today)

1. `make generate`
2. `make stage-images && make stage-models`
3. `./aegis-cli bundle ...`
4. `pulumi up` (creates VM + **Cloud NAT**)
5. `gcloud compute scp` the bundle
6. SSH in, run `bootstrap.sh` (this step installs drivers)
7. Manually delete NAT
8. Run validation

**Limitation**: The VM requires outbound internet for ~10-20 minutes during first boot.

### Phase 4 Flow (recommended for production air-gap)

1. **One-time**: Build golden image with Packer (see below)
2. `make generate --profile gcp-hardened`
3. Create bundle (same as before)
4. `pulumi up` with `use_golden_image: true` (no NAT created)
5. `gcloud scp` the bundle only
6. Bootstrap runs in < 2 minutes
7. Validation — **the VM has never had a public route**

## Phase 4: Building the Golden Image

![Golden Image Baking Pipeline](../images/phase4-golden-image-pipeline.jpg)

### Step-by-Step

```bash
cd scripts/image-bake/packer

packer init .
packer build \
  -var="project_id=YOUR_PROJECT_ID" \
  -var="zone=us-central1-a" \
  gcp-aegis-golden.pkr.hcl
```

This takes 15–25 minutes and produces a custom image family:

`aegis-golden-ubuntu-2204-nvidia-k3s`

The image contains:
- NVIDIA 535 drivers + CUDA
- NVIDIA Container Toolkit (configured for containerd)
- K3s (disabled until bundle arrives)
- Pre-created `/opt/aegis/*` directories + `aegis` user

### Using the Golden Image

Edit your Pulumi stack config:

```bash
cd iac/pulumi
pulumi config set use_golden_image true
pulumi config set custom_image_family aegis-golden-ubuntu-2204-nvidia-k3s
```

Then run:

```bash
pulumi up
```

The resulting VM will boot with **no Cloud NAT** and no external IP access config if you desire.

## Prove the Air Gap — 3 Success Criteria

This is the moment of truth. Use the visual checklist below and the automated script.

![Prove the Air Gap Checklist](../images/prove-airgap-checklist.jpg)

### Automated Validation (run from inside the cluster)

```bash
# After bundle is applied
kubectl -n aegis create -f - <<EOF
apiVersion: v1
kind: Pod
metadata:
  name: validator
spec:
  containers:
  - name: validator
    image: curlimages/curl
    command: ["/bin/sh", "-c", "sleep 3600"]
EOF

kubectl -n aegis cp scripts/validate.sh validator:/validate.sh
kubectl -n aegis exec -it validator -- /validate.sh
```

All three checks must pass **after** you have removed the NAT / external IP.

## Troubleshooting & FAQs

**Q: The golden image build fails on NVIDIA driver installation**
- Make sure the builder VM has a T4 attached (`accelerator_type = "nvidia-tesla-t4"` in the Packer file). NVIDIA drivers refuse to install cleanly without a GPU present during build.

**Q: After switching to golden image, pods are still in ImagePullBackOff**
- You still need to deliver the `.bundle` and run `bootstrap.sh`. The golden image only removes the driver/K3s installation steps.

**Q: How do I know the VM truly had no internet?**
- Best proof: launch the instance with **no external IP** and no Cloud NAT at all (`access_configs: []`). If it boots and inference works, you have succeeded.

**Q: Can I use this with vLLM instead of Ollama?**
- Not in the current profiles, but the architecture supports it. Phase 5 will add vLLM + quantized GGUF/EXL2 options.

**Q: Bundle is too big to scp (~6-8 GB)**
- Recommended: attach a second persistent disk (500 GB) to the instance in Pulumi, format it, and copy the bundle to that disk from a jump host that has internet. Then detach the disk and attach it to the air-gapped VM.
